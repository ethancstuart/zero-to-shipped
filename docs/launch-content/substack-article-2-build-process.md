## Body

The bottleneck moved.

For a decade, the constraint in building software was engineering capacity. You had the idea. You wrote the spec. You waited for a sprint slot that kept getting pushed.

That constraint is gone.

AI coding tools have compressed what used to take a sprint into a conversation. What used to require a team now requires a prompt — and someone who knows what they're actually trying to build.

The pattern I see everywhere right now: the bottleneck shifted from "can we build this?" to "should we build this, and exactly what does this mean?" That question has always been the PM's job. But now the answer has to come with a working prototype, not a PRD.

The PMs who understand this are shipping things their backlog never would have reached. The ones who don't are writing specifications for engineers who are building the same thing in parallel with Cursor.

---

Six months ago I decided to understand this by doing it.

I built Zero to Ship — a course that teaches PMs, BAs, project managers, and BI engineers to build real, deployed software using AI coding tools. Not to become engineers. To close the loop between specification and execution that AI has made possible for the first time.

The curriculum is 16 modules, from setup to a shipped capstone project. Every module produces something real — not an exercise, not a sandbox demo. Something you can point to.

The premise is simple: the skills that make you good at product — writing precise specs, evaluating output, knowing immediately what's wrong with something — are exactly the skills that make AI-assisted building work. You're not starting from zero. You're unlocking what you already have.

---

I built the platform using the same tools I'm teaching with. Cursor. Claude Code. Supabase. Stripe. Every bug fixed by pasting an error into an AI and working through it. Every feature started as a specification, not a prompt.

What I learned: the risk in building didn't disappear. It moved.

It used to be "can we build this?" That risk is largely gone. The risk now is building the wrong thing, or too many of the wrong things. That's a judgment problem, not a technical one. It's the problem product people were already trained to solve — they just didn't have a direct path to execution.

That gap is what Zero to Ship closes.

---

There's a version of this shift that goes badly. PMs start shipping things faster than they're deciding what to ship. They move faster with worse judgment. The backlog that used to get stuck in engineering now gets stuck in a different place — the one between "I can build this in an afternoon" and "I should build this at all."

The goal isn't to make every PM a builder. It's to make every PM understand the loop well enough to lead it.
