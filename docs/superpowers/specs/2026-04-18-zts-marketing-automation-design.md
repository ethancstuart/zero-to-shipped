# ZTS Marketing Automation — Design Spec

**Date:** 2026-04-18  
**Status:** Approved (council-reviewed)  
**Scope:** Standalone automated content engine for Zero to Ship (zerotoship.app)  
**Repo:** `zts-marketing` (new standalone repo, deployed to Vercel)  
**Build estimate:** 2.5–3 days solo

---

## Overview

A fully automated marketing engine that generates and publishes content across LinkedIn, Twitter/X, and Beehiiv newsletter on a fixed cadence. Posts are generated by Claude API based on content pillars and Ethan's voice. Buffer API handles LinkedIn and Twitter/X publishing. Beehiiv API auto-publishes the weekly newsletter.

A thin admin dashboard provides a 24-hour edit window for any post before it goes live, plus a config panel to adjust pillars, cadence, and voice notes without touching code. Push notifications alert when posts enter the edit window. API failures alert immediately.

---

## Architecture

```
Vercel Cron (daily 6am UTC)
  → Claude API (generate posts)
  → Vercel KV (store post log + config)
  → Buffer API (queue LinkedIn + Twitter/X posts)
  → Beehiiv API (auto-publish weekly newsletter)

Admin UI (zts-marketing.vercel.app)
  → reads Vercel KV (upcoming posts, config)
  → edit/delete posts within 24h window
  → Config panel (pillars, schedule, voice notes, pause, positioning anchor)
  → Push notifications via Resend (edit window alerts + API failure alerts)
```

**Isolation:** Zero coupling to `zerotoship.app`. Separate Vercel project, separate env vars, separate KV namespace.

## Content Pillars

| Pillar | Target % | Voice frame | CTA | Goal |
|---|---|---|---|---|
| Build-Story | 40% | PM primary, maker modifier | `/preview/module-1` | Trust + shareability |
| Method | 25% | PM primary | `/pricing` | Email opt-ins + saves |
| Outcome | 20% | PM primary | `/preview/module-1` | Conversion |
| Product | 15% | Product-first | `/pricing` | Awareness |

**Voice frame clarification:** Ethan's primary identity across all pillars is PM ("I'm a product manager who builds things"). For Build-Story posts, the maker angle is a modifier on that identity ("I'm a PM — and I spent the weekend building something"). Never two alternate personas; one coherent person in different contexts.

**Pillar rotation:** Before each generation, the cron reads the last 14 days of published posts from Vercel KV. It calculates which pillar is underweight vs. target and assigns it to the next post. The same pillar is never used on consecutive LinkedIn posts.

## Publishing Cadence

| Channel | Frequency | Schedule (UTC) |
|---|---|---|
| LinkedIn (personal profile) | 2x/week | Tue + Fri, 12:00 UTC (8am ET) — weekdays only |
| Twitter/X (personal handle) | 2x/day | Daily 13:00 UTC + 19:00 UTC (90-min gap enforced) |
| Beehiiv newsletter | 1x/week | Sunday 13:00 UTC, auto-published |

All times stored and computed in UTC. LinkedIn never posts on weekends (algorithm deprioritizes weekend posts). Twitter/X secondary slot at 19:00 UTC doubles reach at zero marginal generation cost — alternates between single tweet (Build-Story) and 3-tweet thread.

LinkedIn and Twitter/X publish via Buffer OAuth connected to Ethan's personal accounts. Beehiiv publishes directly via API to the full subscriber list.

## Content Generation Engine

### System Prompt Architecture

Each generation call to Claude receives:

1. **Static context (system prompt):**
   - Ethan's voice rules: first-person singular ("I", never "we"), conversational, no jargon, no buzzwords, no passive voice
   - Primary identity: PM who builds things. Maker angle is a modifier for Build-Story, not an alternate persona.
   - Positioning anchor (from `config:positioningAnchor`): two sentences — the core claim and the core enemy — injected verbatim into every system prompt so every post ladders back to the brand
   - Hard constraints (see Voice + Brand Constraints section)
   - ZTS product context: what it is, who it's for, pricing, key URLs
   - Brand voice: "showing work" not self-promotion; outcome-led not feature-led
   - CTA by pillar: Build-Story + Outcome → `/preview/module-1`; Method + Product → `/pricing`

2. **Dynamic context (per generation):**
   - Assigned pillar for this post
   - Target channel (shapes format and length)
   - Last 7 published posts on this channel with their opening lines (deduplication + semantic similarity check)
   - Active hashtag pool with last-used dates (LinkedIn only)
   - Today's date in UTC (for timing references)
   - Voice notes from `config:voiceNotes` (freeform steering, e.g. "launching April 28", "founding spots filling up")

### Output Format

Claude returns structured JSON:

```json
{
  "content": "post body text",
  "channel": "linkedin | twitter | beehiiv",
  "pillar": "build-story | method | outcome | product",
  "slot": "primary | secondary",
  "scheduled_at": "ISO 8601 UTC timestamp",
  "pinned_comment": "CTA text for LinkedIn (null for other channels)",
  "thread": ["tweet 1", "tweet 2", "tweet 3"],
  "hashtags_used": ["tag1", "tag2"]
}
```

`thread` is null for single tweets. `hashtags_used` is null for non-LinkedIn posts.

### LinkedIn Format Rules
- 150–280 words
- No external links in body — link goes in `pinned_comment`, posted immediately after via Buffer
- Hashtags: exactly 2, selected from `config:hashtagPool`, no tag repeated within a 5-post rolling window; tags appended at end only
- Ends with a question or implicit hook — never a CTA in the body
- Pinned comment CTA is pillar-keyed:
  - Build-Story / Outcome → `→ Try Module 1 free: zerotoship.app/preview/module-1`
  - Method / Product → `→ See what's inside: zerotoship.app/pricing`
- Semantic dedup: opening line must not semantically duplicate any headline from the last 7 LinkedIn posts

### Twitter/X Format Rules
- Single tweet: under 280 chars
- Method pillar: 3-tweet thread (opener → content → CTA)
- Secondary 19:00 UTC slot: alternates Build-Story single tweet and Method thread
- 90-minute gap enforced between any two posts on the same channel (cron-level guard, not just prompt-level)
- No promotional language in opener — hook first, product second

### Beehiiv Newsletter Format
- Weekly, ~400–600 words
- Build-Story or Method pillar (alternates weekly)
- Includes one product mention and one pillar-keyed CTA
- Fully auto-published via Beehiiv API on Sunday 13:00 UTC

## Hashtag Pool

Maintained in `config:hashtagPool`. Rotation enforced: no tag repeats within a 5-post rolling window on LinkedIn. Starter pool (editable via Config panel):

```
#ProductManagement #BuildInPublic #AITools #ProductManager
#NoCode #LowCode #PMLife #SideProject #ShipIt
#CareerGrowth #LearnToCode #AIProductivity
```

Generator receives the pool + last-used dates. It selects 2 tags that (a) fit the post topic and (b) haven't appeared in the last 5 LinkedIn posts.

## Lead Magnet + Beehiiv Opt-In

To drive net-new Beehiiv subscribers from social (not just publish to the existing list):

- **Asset:** "PM AI Toolkit" — one-page cheat sheet of the AI build loop, tools, and prompts from ZTS Module 1. Standalone value, no course purchase required.
- **Landing page:** `zerotoship.app/toolkit` — email opt-in → Beehiiv subscriber → sends toolkit PDF via Resend → adds to ZTS newsletter list
- **Social integration:** Build-Story and Method posts include a secondary note in the pinned comment: `Free toolkit → zerotoship.app/toolkit`
- **Admin config:** Toggle in Config panel (`config:leadMagnetActive`) — when enabled, pinned comments include the toolkit CTA. When disabled (e.g., during founding-spots push), omit it.

This is a V1 feature. The toolkit PDF is a content task for Ethan; the opt-in page and Beehiiv integration is a build task.

## Vercel KV Schema

All times stored in UTC. No local timezone assumptions anywhere in the codebase.

```
posts:{id}              → { content, channel, pillar, slot, scheduled_at_utc, status, pinned_comment, hashtags_used, created_at }
posts:index             → sorted set of post IDs by scheduled_at_utc
config:pillars          → { buildStory: 40, method: 25, outcome: 20, product: 15 }
config:schedule         → {
                            linkedin: { days: [2, 5], time_utc: "12:00" },
                            twitter_primary: { time_utc: "13:00" },
                            twitter_secondary: { time_utc: "19:00" },
                            beehiiv: { day: 0, time_utc: "13:00" }
                          }
config:channels         → { linkedin: true, twitter: true, beehiiv: true }
config:paused           → boolean
config:pauseUntil       → ISO 8601 UTC timestamp (optional — for time-limited pauses e.g. breaking news)
config:voiceNotes       → string, max 500 chars (hard limit — longer input silently truncated)
config:positioningAnchor → string (2 sentences: core claim + core enemy — injected into every system prompt)
config:hashtagPool      → [{ tag: string, lastUsedPostIndex: number }]
config:leadMagnetActive → boolean
```

Post statuses: `pending` → `queued_buffer` → `published` | `deleted` | `failed`

## Admin UI

**URL:** `zts-marketing.vercel.app` (Vercel password protection — no custom auth to build)

### Content Calendar (default view)

- Lists all upcoming posts sorted by `scheduled_at_utc`
- **Blue border** = `status == "pending" AND scheduled_at_utc - now ≤ 24h` — editable and deletable
- **Grey** = future posts (`status == "pending" AND scheduled_at_utc - now > 24h`) — view only
- **Red border** = `status == "failed"` — requires manual intervention
- **Yellow border** = stale: `status == "queued_buffer" AND scheduled_at_utc < now - 48h` — likely a Buffer delivery failure
- Each post shows: channel badge, pillar badge, slot badge (primary/secondary), UTC publish time, post body preview, Edit + Delete buttons (blue border only)

### Edit Post
- Full-text edit of post body
- Edit pinned comment (LinkedIn only)
- Save updates KV only (status is always `pending` at edit time — posts are never in Buffer yet when editable)

### Alerts + Notifications
- **Edit window alert (push):** When a post enters the 24h edit window (Phase 1 completes), Resend sends an email to Ethan with the post body. No browser required — he can judge from his inbox.
- **API failure alert (push):** Any 4xx/5xx from Buffer, Claude, or Beehiiv triggers a Resend email within the same cron run. Email includes: which API failed, which post was affected, current post status.
- **Stale alert:** Admin UI banners any post in yellow-border state. No push notification — visible on next admin visit.
- **Buffer OAuth warning:** If Buffer returns a 401, the alert email specifically says "Buffer OAuth expired — re-authorize at buffer.com" (not just "API error").

### Config Panel
- **Pillar weights:** sliders, auto-normalize to 100% on save (no error state — just redistribute proportionally), updates `config:pillars`
- **Positioning anchor:** textarea (2 sentences), updates `config:positioningAnchor` — injected into every generation
- **Hashtag pool:** tag list with add/remove, updates `config:hashtagPool`
- **Channel toggles:** on/off per channel, updates `config:channels`
- **Schedule:** UTC time pickers per channel slot, updates `config:schedule`
- **Voice notes:** freeform textarea, 500 char max with counter, updates `config:voiceNotes`
- **Lead magnet toggle:** on/off, updates `config:leadMagnetActive`
- **Pause all:** toggle `config:paused`
- **Pause until:** datetime picker for time-limited pause (breaking news use case), sets `config:pauseUntil`

*History View: out of scope for V1. Add post-launch.*

## Cron Flow (Daily, 6am UTC)

The cron does three things on every run: generate tomorrow's posts, push today's posts to Buffer, mark published posts.

```
PHASE 1 — Generate (for tomorrow's schedule)
1. Check config:paused OR config:pauseUntil > now → if true, exit all phases
2. Read config:channels → determine which channels are scheduled for tomorrow
3. For each channel/slot due tomorrow:
   a. Read last 14 days of published posts from KV (pillar deduplication)
   b. Read last 7 posts on this channel for semantic dedup (pass opening lines to Claude)
   c. Read config:pillars → calculate underweight pillar
   d. Read config:hashtagPool with last-used dates (LinkedIn only)
   e. Read config:voiceNotes, config:positioningAnchor
   f. Call Claude API with assembled context
   g. On Claude failure (timeout, malformed JSON, rate limit): log error → send Resend alert → skip this slot (do not retry — next cron run will catch it)
   h. On success: write post to KV with status: "pending", scheduled_at_utc = tomorrow at slot time

PHASE 2 — Push to Buffer (posts entering their 24h window)
4. Query KV for posts with status: "pending" AND scheduled_at_utc ≤ now + 24h
5. For each post:
   a. Idempotency check: if a post with this id is already in "queued_buffer", skip (prevents duplicate pushes on cron retry)
   b. 90-minute gap check: if another post on the same channel is already in "queued_buffer" with scheduled_at_utc within 90 minutes of this post, skip with a warning log
   c. POST to Buffer API
   d. On Buffer 401: log → send Resend alert ("Buffer OAuth expired") → mark post status: "failed" → exit Phase 2
   e. On other Buffer error: log → send Resend alert → mark post status: "failed" → continue to next post
   f. On success: update KV status to "queued_buffer"

PHASE 3 — Mark published
6. Query KV for posts with status: "queued_buffer" AND scheduled_at_utc < now
7. Mark as "published" (Buffer does not send webhooks on free/basic plans — polling only)
8. Stale check: any post with status "queued_buffer" AND scheduled_at_utc < now - 48h → log warning (surfaced as yellow border in admin UI)
```

**Empty history fallback (first run):** If KV has no post history, pillar defaults to Build-Story. Semantic dedup context is an empty array.

**Edit window definition:** Between Phase 1 (6am D-1) and Phase 2 (6am D-0) — approximately 24 hours. Blue border = `status == "pending" AND scheduled_at_utc - now ≤ 24h`.

## Voice + Brand Constraints (Hard-coded in System Prompt)

- **Never mention:** Disney, current employer, professional affiliation, clients, internal projects
- **Never use:** corporate language, buzzwords, passive voice, emoji of any kind
- **Never tag or mention:** any individual person or account
- **Never publish:** links in LinkedIn post body
- **Never repeat:** same opening word or phrase as any post published in the last 14 days on this channel
- **Always use:** first-person singular ("I", not "we")
- **Primary identity:** PM who builds things. Maker angle is a modifier for Build-Story, not an alternate persona.
- **No-ad test:** every post must pass — "would a thoughtful person share this if they'd never heard of ZTS?"
- **Positioning anchor:** always reference `config:positioningAnchor` — every post ladders back to the core claim

This system is being built **before** the April 28 ZTS launch. The April 28 launch posts (already drafted in `docs/launch-content/`) are still executed manually — this engine takes over from April 29 onward.

**Build estimate:** 2.5–3 days solo. Do not start fewer than 4 days before April 28.

- [ ] Beehiiv API key configured, second ZTS publication created
- [ ] Claude API key in env vars

- [ ] `config:positioningAnchor` written (2 sentences: core claim + core enemy)
