# ZTS Marketing Automation — Design Spec

**Date:** 2026-04-18  
**Status:** Approved  
**Scope:** Standalone automated content engine for Zero to Ship (zerotoship.app)  
**Repo:** `zts-marketing` (new standalone repo, deployed to Vercel)

---

## Overview

A fully automated marketing engine that generates and publishes content across LinkedIn, Twitter/X, and Beehiiv newsletter on a fixed cadence. Posts are generated by Claude API based on content pillars and Ethan's voice. Buffer API handles LinkedIn and Twitter/X publishing. Beehiiv API auto-publishes the weekly newsletter.

A thin admin dashboard provides a 24-hour edit window for any post before it goes live, plus a config panel to adjust pillars, cadence, and voice notes without touching code.

---

## Architecture

```
Vercel Cron (daily 6am UTC)
  → Claude API (generate posts)
  → Vercel KV (store post log + config)
  → Buffer API (queue LinkedIn + Twitter/X posts)
  → Beehiiv API (auto-publish weekly newsletter)

Admin UI (zts-marketing.vercel.app)
  → reads Vercel KV (upcoming posts, config)
  → edit/delete posts within 24h window
  → Config panel (pillars, schedule, voice notes, pause)
```

**Isolation:** Zero coupling to `zerotoship.app`. Separate Vercel project, separate env vars, separate KV namespace.

## Content Pillars

| Pillar | Target % | Voice frame | Goal |
|---|---|---|---|
| Build-Story | 40% | Maker ("I build things") | Trust + shareability |
| Method | 25% | PM ("I'm a product manager") | Email opt-ins + saves |
| Outcome | 20% | PM ("I'm a product manager") | Conversion |
| Product | 15% | Product-first | Awareness |

**Pillar rotation:** Before each generation, the cron reads the last 14 days of published posts from Vercel KV. It calculates which pillar is underweight vs. target and assigns it to the next post. The same pillar is never used on consecutive LinkedIn posts.

## Publishing Cadence

| Channel | Frequency | Default schedule |
|---|---|---|
| LinkedIn (personal profile) | 2x/week | Tuesday + Friday, 8:00am ET |
| Twitter/X (personal handle) | 1x/day | Daily, 9:00am ET |
| Beehiiv newsletter | 1x/week | Sunday, auto-published |

LinkedIn and Twitter/X publish via Buffer OAuth connected to Ethan's personal accounts. Beehiiv publishes directly via API to the full subscriber list.

## Content Generation Engine

### System Prompt Architecture

Each generation call to Claude receives:

1. **Static context (system prompt):**
   - Ethan's voice rules: first-person, conversational, no jargon
   - Persona switching: PM framing for Method/Outcome pillars; maker framing for Build-Story pillars
   - Hard constraints: never mention Disney, current employer, or professional affiliation; never include links in LinkedIn post body (pinned comment only); Twitter/X defaults to single tweet, Method pillar generates 3-tweet thread
   - ZTS product context: what it is, who it's for, pricing, key URLs
   - Brand voice: "showing work" not self-promotion; outcome-led not feature-led

2. **Dynamic context (per generation):**
   - Assigned pillar for this post
   - Target channel (shapes format and length)
   - Last 3 published posts on this channel (deduplication)
   - Today's date (for timing references)
   - Voice notes from Config panel (freeform steering text, e.g. "launching April 28", "founding spots filling up")

### Output Format

Claude returns structured JSON:

```json
{
  "content": "post body text",
  "channel": "linkedin | twitter | beehiiv",
  "pillar": "build-story | method | outcome | product",
  "scheduled_at": "ISO 8601 timestamp",
  "pinned_comment": "link text for LinkedIn (null for other channels)",
  "thread": ["tweet 1", "tweet 2", "tweet 3"] // Twitter/X Method pillar only
}
```

### LinkedIn Format Rules
- 150–280 words
- No external links in body — link goes in `pinned_comment`, posted immediately after the main post via Buffer
- No hashtag stuffing — max 2 relevant hashtags, at end only
- Ends with a question or implicit hook (never a CTA)
- CTA lives in pinned comment: `→ Try Module 1 free: zerotoship.app/preview/module-1`

### Twitter/X Format Rules
- Single tweet: under 280 chars
- Method pillar: 3-tweet thread (opener → content → CTA)
- No promotional language in opener — hook first, product second

### Beehiiv Newsletter Format
- Weekly, ~400–600 words
- Build-Story or Method pillar (alternates weekly)
- Includes one product mention and one CTA to `zerotoship.app/preview/module-1`
- Fully auto-published via Beehiiv API on Sunday

## Vercel KV Schema

```
posts:{id}          → { content, channel, pillar, scheduled_at, status, pinned_comment, created_at }
posts:index         → sorted set of post IDs by scheduled_at
config:pillars      → { buildStory: 40, method: 25, outcome: 20, product: 15 }
config:schedule     → { linkedin: { days: [2,5], time: "12:00" }, twitter: { time: "13:00" }, beehiiv: { day: 0, time: "13:00" } }
config:channels     → { linkedin: true, twitter: true, beehiiv: true }
config:paused       → boolean
config:voiceNotes   → string (freeform, read on each generation)
```

Post statuses: `pending` → `queued_buffer` → `published` | `deleted`

## Admin UI

**URL:** `zts-marketing.vercel.app` (Vercel password protected)

### Content Calendar (default view)

- Lists all upcoming posts sorted by `scheduled_at`
- **Blue border** = within 24h edit window — editable and deletable
- **Grey** = future posts, stored in KV but not yet pushed to Buffer
- Each post shows: channel badge, pillar badge, time-to-publish, post body preview, Edit + Delete buttons
- Alert banner when any post is in the edit window

### Edit Post
- Full-text edit of post body
- Edit pinned comment (LinkedIn only)
- Save pushes updated content to KV (and to Buffer if already queued)

### Config Panel
- **Pillar weights:** sliders, must sum to 100%, updates `config:pillars`
- **Channel toggles:** on/off per channel, updates `config:channels`
- **Schedule:** day/time pickers per channel, updates `config:schedule`
- **Voice notes:** freeform textarea, updates `config:voiceNotes` — read on every generation run
- **Pause all:** toggle that sets `config:paused = true`, stops cron from pushing to Buffer

### History View
- Last 30 published posts with channel, pillar, publish timestamp

## Cron Flow (Daily, 6am UTC)

The cron does two things on every run: generate tomorrow's posts, then push today's posts to Buffer.

```
PHASE 1 — Generate (for tomorrow's schedule)
1. Read config:paused → if true, exit both phases
2. Read config:channels → determine which channels are scheduled for tomorrow
3. For each channel due tomorrow:
   a. Read last 14 days of published posts from KV (pillar deduplication)
   b. Read config:pillars → calculate underweight pillar
   c. Read config:voiceNotes
   d. Call Claude API with assembled context
   e. Parse JSON response
   f. Write post to KV with status: "pending", scheduled_at = tomorrow at channel default time

PHASE 2 — Push to Buffer (posts scheduled within next 24h)
4. Query KV for all posts with status: "pending" and scheduled_at ≤ now + 24h
5. For each: POST to Buffer API → update KV status to "queued_buffer"

PHASE 3 — Mark published
6. Query KV for posts with status: "queued_buffer" and scheduled_at < now
7. Mark as "published" (Buffer does not send webhooks on free/basic plans — polling instead)
```

**Edit window:** Between Phase 1 (6am D-1) and Phase 2 (6am D-0) — approximately 24 hours. Admin UI shows blue border on any post not yet in `queued_buffer` status.

## Voice + Brand Constraints (Hard-coded in System Prompt)

- **Never mention:** Disney, current employer, professional affiliation, clients, internal projects
- **Never use:** corporate language, buzzwords, passive voice
- **Never publish:** links in LinkedIn post body
- **Always use:** first-person singular ("I", not "we")
- **Persona by pillar:**
  - Build-Story → "I build things on the side"
  - Method/Outcome → "I'm a product manager"
  - Product → product-as-subject (no persona needed)

## Launch Path

This system is being built **before** the April 28 ZTS launch, replacing the manual sprint plan. Once running, the engine handles all post-launch content sustainably. The April 28 launch posts (already drafted in `docs/launch-content/`) are still executed manually — this engine takes over from April 29 onward.

**Pre-launch checklist:**
- [ ] `zts-marketing` repo created
- [ ] Buffer account connected (LinkedIn + Twitter/X OAuth)
- [ ] Beehiiv API key configured
- [ ] Claude API key in env vars
- [ ] Vercel KV provisioned
- [ ] Admin UI deployed + password protected
- [ ] First batch of posts generated and reviewed
- [ ] Cron verified in Vercel dashboard
