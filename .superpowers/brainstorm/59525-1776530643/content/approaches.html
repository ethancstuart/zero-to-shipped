3 Approaches — ZTS Marketing Automation

All three use Buffer API for LinkedIn + Twitter/X. Pick the architecture that fits how much you want to be involved.

A

Pure Cron (Vercel) — Full Autopilot

How it works: A standalone zts-marketing Vercel project. Cron jobs fire daily → Claude generates posts from your content pillars → Buffer API queues them for LinkedIn + Twitter/X. Zero UI. Set it and forget it.

Build time: 1–1.5 days

Pros

  • Fastest to ship
  • Zero ongoing effort
  • No hosting cost (Vercel free tier)

Cons

  • No visibility into what's going out
  • No override if Claude writes something off
  • Hard to tune without touching code
B

Thin Admin UI — Automated with Emergency Brake ⭐ Recommended

How it works: Same cron engine as A, but adds a lightweight admin dashboard. Posts auto-queue to Buffer 24h before they go live. Dashboard shows upcoming posts — you can edit or delete any post in that window. If you don't touch it, it publishes automatically.

Build time: 2–2.5 days

Pros

  • Still fully automated by default
  • Emergency brake if something goes sideways
  • Content calendar visible at a glance
  • Easy to tune pillars via UI

Cons

  • 1 extra day of build
  • One more thing to deploy + maintain
C

GitHub Actions + Buffer API — Developer-Native

How it works: Cron jobs run in GitHub Actions (free). Script generates a batch of posts, opens a PR with the content. You merge to approve the batch, GitHub Action then pushes to Buffer queue. Review is optional — you can auto-merge via a label.

Build time: 1–1.5 days

Pros

  • No hosting cost at all
  • Git history of all content
  • PR review flow is familiar

Cons

  • GitHub-y — not a product feel
  • PR merge = extra friction
  • Less polished for a course founder persona

My recommendation: B — you get full automation with a safety net. The extra day of build is worth it given posts go out under your name.
